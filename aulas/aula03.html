inteli

Entidades: as "coisas" que o sistema precisa conhecer

Uma entidade é qualquer elemento do domínio que tem identidade própria, estado observável e comportamento relevante para o sistema. No código OO, cada entidade vira uma classe.

📌 Como identificar uma entidade

  • É uma "coisa" sobre a qual o sistema precisa guardar informação?
  • Tem atributos que a distinguem de outras do mesmo tipo?
  • Tem comportamentos que só ela executa?
  • Existe independentemente de outras entidades?
  • Aparece como substantivo na descrição do problema?

🎭 Entidades do Portal Aventura

  • Espetáculo — tem nome, gênero, elenco, duração
  • Usuário — tem perfil, histórico, preferências
  • Teatro — tem nome, endereço, capacidade
  • Sessão — liga um espetáculo a um teatro em uma data
  • Ingresso — representa a compra de acesso a uma sessão
  • Recomendação — resultado do algoritmo para um usuário

Entidade ≠ tabela de banco de dados

Um erro comum é confundir entidade OO com tabela relacional. Em OO, uma entidade pode conter listas, objetos embutidos e comportamento. Em SQL, a estrutura é normalizada em linhas e colunas. O diagrama UML descreve o domínio; o modelo relacional descreve o armazenamento. São camadas diferentes do sistema.

Atributos e métodos: o que o objeto É e o que ele FAZ

Toda classe tem dois tipos de membros: atributos (estado interno — o que o objeto sabe sobre si mesmo) e métodos (comportamento — o que o objeto pode fazer ou calcular).

class Espetaculo:
    """Representa um espetáculo no catálogo do Portal Aventura."""

    def __init__(self, nome: str, genero: str,
                 duracao_min: int, sinopse: str):
        # Atributos: o que o objeto sabe sobre si mesmo
        self._nome         = nome          # privado: acesso via getter
        self._genero       = genero
        self._duracao_min  = duracao_min
        self._sinopse      = sinopse
        self._sessoes: list = []           # lista de Sessao
        self._tags:   list = []            # list[str]

    # --- Getters (encapsulamento) ---
    @property
    def nome(self) -> str:
        return self._nome

    @property
    def genero(self) -> str:
        return self._genero

    # --- Métodos: o que o objeto FAZ ---
    def adicionar_sessao(self, sessao) -> None:
        """Vincula uma Sessao a este espetáculo."""
        self._sessoes.append(sessao)

    def calcular_preco_medio(self) -> float:
        """Retorna a média de preço entre todas as sessões."""
        if not self._sessoes:
            return 0.0
        return sum(s.preco for s in self._sessoes) / len(self._sessoes)

    def obter_tags(self) -> list:
        """Retorna a lista de tags associadas ao espetáculo."""
        return list(self._tags)

    def __repr__(self) -> str:
        return f"Espetaculo({self._nome!r}, {self._genero!r})"

O que observar no código

  1. Atributos com _: prefixo de convenção Python para indicar acesso restrito. Não é acesso privado de verdade (Python não bloqueia), mas sinaliza que não se deve acessar de fora da classe.
  2. @property: getter que expõe o valor sem permitir escrita direta. Para alterar, usaria @nome.setter com validação.
  3. Métodos calculados: calcular_preco_medio() não armazena o preço médio — calcula quando chamado. Evita inconsistência entre estado armazenado e dados reais.
  4. Listas internas: _sessoes é gerenciada pela própria classe. O método obter_tags() retorna uma cópia para evitar que o chamador modifique a lista original.

Encapsulamento: por que não expor tudo como público

Encapsulamento é o princípio de esconder o estado interno de um objeto e controlar como ele é acessado e modificado. Não é sobre segurança — é sobre manutenibilidade: quando os detalhes internos mudam, o código externo não quebra.

🔒 Visibilidade em UML

+ público: qualquer classe pode acessar diretamente.

- privado: somente a própria classe acessa.

# protegido: a classe e suas subclasses acessam.

⚠️ O problema de tudo público

Se qualquer código pode alterar espetaculo.preco = -50 diretamente, não há validação possível. Com encapsulamento, o setter garante: if preco < 0: raise ValueError. O contrato da classe é preservado.

🎭 Exemplo: Usuario no Portal

_historico é privado. Só o método registrar_visualizacao() o atualiza, com validação de duplicata e timestamp automático. Código externo nunca manipula a lista diretamente, garantindo consistência.

Regra prática para modelagem

Comece com tudo privado (- em UML). Torne público apenas o que outros precisam acessar, e apenas através de métodos com intenção clara. Se você está expondo um atributo diretamente, pergunte: "esse acesso pertence à interface pública desta classe ou é um detalhe de implementação?"

Notação UML de classes: como ler e escrever um diagrama

Um diagrama de classes UML tem uma notação padronizada. Ler qualquer diagrama UML no mercado requer conhecer exatamente esses elementos.

Espetáculo
- nome: String
- genero: String
- duracao: Integer
- sinopse: String
- sessoes: List<Sessao>
+ getNome(): String
+ adicionarSessao(s: Sessao): void
+ calcularPrecoMedio(): Float
+ obterTags(): List<String>

↑ exemplo de classe UML completa

Lendo o diagrama

  1. Compartimento 1 (nome): nome da classe em PascalCase. Interfaces levam «interface»; classes abstratas ficam em itálico.
  2. Compartimento 2 (atributos): formato visibilidade nome: Tipo. O tipo é da linguagem-alvo ou genérico (String, Integer, List).
  3. Compartimento 3 (métodos): formato visibilidade nome(params): ReturnType. void significa que não retorna valor.
  4. Visibilidade: + público, - privado, # protegido.
  5. Multiplicidade: aparece nas linhas de relacionamento. 1..* significa "um ou mais"; 0..1 significa "zero ou um".

Mapeamento UML do Portal Aventura

Do TAPI ao diagrama de classes UML

Dinâmica · em grupo · 40 min

Do TAPI ao diagrama de classes UML

  1. Releia o TAPI e identifique entidades (10 min): com base na descrição do Portal Aventura, listem de 6 a 8 entidades do domínio. Para cada uma, respondam: o que ela sabe sobre si mesma (atributos) e o que ela faz (métodos)? Escrevam em texto antes de qualquer diagrama.
  2. Separe em contextos (5 min): agrupem as entidades em 2 ou 3 contextos distintos. Cada contexto deve ter uma pergunta de negócio que só ele responde. Escrevam essa pergunta ao lado do contexto.
  3. Construa o diagrama UML (20 min): usando papel, quadro ou ferramenta digital (Lucidchart, draw.io, Mermaid), desenhem o diagrama de classes completo. Para cada relação, escolham explicitamente o tipo: associação, agregação, composição ou herança — e escrevam o porquê ao lado.
  4. Defenda uma decisão (5 min): cada grupo escolhe uma decisão de modelagem que gerou discussão interna e apresenta em 2 minutos: qual era a dúvida, quais opções foram consideradas e por que o grupo escolheu o caminho adotado.

Foco: o sistema de recomendação. Priorizem as classes que fazem a recomendação funcionar antes das classes comerciais. Se ficarem em dúvida sobre onde colocar uma classe, perguntem: "em qual contexto ela mudaria com mais frequência?" e a coloquem lá.

Debrief: o que costuma aparecer na primeira modelagem

Três padrões que aparecem em todo grupo que modela um domínio real pela primeira vez.

1 · Onde fica a Recomendação?

Grupos frequentemente colocam recomendar() dentro de Usuário ou de Espetáculo. O problema: essas classes ficam acopladas ao algoritmo. A melhor separação é ter um contexto de Recomendação com Algoritmo e Recomendação como classes próprias, que leem dados de Usuário e Catálogo mas não os alteram.

2 · Composição vs. Agregação

A dúvida mais comum: Sessão é composta de Espetáculo ou associada? Resposta: associação — o espetáculo existe antes da sessão e continua existindo após. Já Compra ◆── Ingresso é composição clara: o ingresso faz sentido apenas como parte da compra. Perguntar "o B existe sem o A?" resolve 90% dos casos.

3 · Herança vs. atributo

Criar EspetáculoMusical e EspetáculoInfantil como subclasses parece natural. Mas se a única diferença é um campo tipo: String, herança é overkill — cria rigidez sem ganho real. Prefira herança quando os subtipos têm comportamentos diferentes, não apenas dados diferentes.

O diagrama é uma conversa, não uma verdade absoluta

Dois times podem produzir diagramas diferentes para o mesmo problema, e os dois podem estar certos, dependendo das prioridades do sistema. O que importa é que o diagrama seja consistente com as regras de negócio, compreensível pela equipe e orientador das decisões de código. Um diagrama que ninguém consulta não serve para nada.

Quiz e checklist

inteli

Checagem 1 de 2 · Relacionamentos UML

No Portal Aventura, um Usuário possui um Perfil que armazena suas preferências de gênero e de histórico de navegação. Se o usuário for deletado do sistema, o perfil dele também deve ser removido, pois ele não faz sentido sem o usuário. Qual relacionamento UML representa corretamente essa situação?

  • Associação simples (linha sólida com seta), porque Usuário referencia Perfil como atributo
  • Agregação (losango vazio em Usuário), porque Perfil pode existir independentemente em outro contexto
  • Composição (losango cheio em Usuário), porque Perfil não tem existência própria fora do Usuário
  • Herança (seta fechada de Perfil para Usuário), porque Perfil é um tipo especializado de Usuário

Checagem 2 de 2 · Separação de contextos

A equipe de engenharia colocou o método recomendar(usuario) diretamente na classe Espetáculo. Qual é o principal problema dessa decisão de modelagem?

  • O nome do método está errado — deveria ser gerarRecomendacao() para seguir a convenção UML

  • A classe Espetáculo passou a conhecer e depender da classe Usuário, criando acoplamento entre contextos que deveriam ser independentes
  • Não há problema — é válido colocar qualquer método em qualquer classe em OO

Checklist de hoje: o que um diagrama UML mínimo tem

Cinco critérios para avaliar se o diagrama do grupo está pronto:

1 · Entidades identificadas

Pelo menos 6 classes modeladas com nome significativo em PascalCase, pertinentes ao domínio do Portal Aventura.

2 · Atributos tipados

Cada classe tem no mínimo 3 atributos com tipo declarado e visibilidade marcada (+, -, #). Sem atributos genéricos sem tipo.

3 · Métodos assinados

Cada classe tem pelo menos 2 métodos com parâmetros e tipo de retorno. Métodos nomeados como verbos de ação.

4 · Relacionamentos corretos

Ao menos 3 tipos distintos de relacionamento usados (associação, composição, agregação ou herança) com justificativa documentada para cada escolha.

5 · Contextos separados

Classes agrupadas em ao menos 2 contextos com responsabilidades distintas. Cada contexto responde a uma pergunta de negócio diferente.

Entrega

Imagem ou link do diagrama UML + documento com a justificativa de pelo menos uma decisão de relacionamento (composição vs. agregação ou herança vs. atributo) + resposta à pergunta: "qual contexto do sistema ficaria mais afetado se os algoritmos de recomendação mudassem e por quê?"
