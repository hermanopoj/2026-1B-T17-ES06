Filtragem baseada em conteúdo vs. filtragem colaborativa

Dois paradigmas fundamentais, dois tipos de dados necessários. A distinção está na origem da informação usada para recomendar: os atributos do item ou o comportamento coletivo dos usuários.

🏷️ Filtragem Baseada em Conteúdo

"Porque você curtiu esse artigo sobre machine learning aplicado à saúde, recomendamos outros sobre deep learning, análise clínica e NLP médico."

  • Usa atributos dos itens (gênero, tags, descrição, autores)
  • Não precisa de outros usuários para funcionar
  • Tende a recomendar itens muito similares (pouca diversidade)
  • Não captura o gosto implícito coletivo

👥 Filtragem Colaborativa

"Usuários que tiveram comportamento similar ao seu também interagiram com esses itens, e você ainda não os viu."

  • Usa o padrão de interações dos usuários (ratings, cliques)
  • Descobre preferências que não estão nos metadados
  • Pode trazer diversidade e surpresas positivas
  • Sofre com cold start: usuário/item novo sem histórico

Sistemas modernos (Netflix, Spotify) usam abordagens híbridas que combinam os dois paradigmas. Desta forma, entender cada um separadamente é pré-requisito para entender o híbrido. O tutorial do RealPython que vocês leram foca exclusivamente em filtragem colaborativa, que é o foco da instrução de hoje.

A matriz usuário-item: o coração da filtragem colaborativa

Toda filtragem colaborativa começa com a mesma estrutura: uma matriz na qual as linhas representam os usuários, as colunas os itens, e cada célula contém a avaliação (ou ausência dela) que o usuário fez daquele item. O problema de recomendação é essencialmente um problema de completar essa matriz.

# Exemplo: matriz usuário-item com ratings 1-5
# NaN = item não avaliado pelo usuário

import pandas as pd
import numpy as np

data = {
    'Filme A': [5.0, 4.0, np.nan, 1.0, np.nan],
    'Filme B': [np.nan, 4.0, 3.0, 2.0, np.nan],
    'Filme C': [3.0, np.nan, np.nan, 5.0, 4.0],
    'Filme D': [np.nan, 5.0, 4.0, np.nan, 3.0],
    'Filme E': [4.0, np.nan, 5.0, np.nan, np.nan],
}
index = ['Ana', 'Bruno', 'Carla', 'Diego', 'Eva']

matriz = pd.DataFrame(data, index=index)
print(matriz)

# Esparsidade
n = matriz.size
preenchidos = matriz.notna().sum().sum()
print(f'\nEsparsidade: {1 - preenchidos/n:.0%}')
# → Esparsidade: 52%

O que cada parte representa

  1. Linha = usuário: o vetor de preferências de um usuário é uma linha da matriz. Usuários similares têm linhas similares.
  2. Coluna = item: cada coluna representa um item. Itens populares têm menos NaN.
  3. NaN = desconhecido: não significa que o usuário não gostou; significa que ele ainda não o viu. É o que queremos prever.
  4. Esparsidade: em sistemas reais, 95–99% das células são NaN. Um usuário médio do Netflix avaliou menos de 0,01% do catálogo. Isso é o principal desafio computacional.

Ratings explícitos (1–5) são o caso mais simples. Dados implícitos (cliques, tempo) são mais comuns em produção mas mais difíceis de modelar.

Esparsidade, cold start e escalabilidade: inimigos da filtragem colaborativa

Entender esses limites é o que permite escolher a abordagem certa para cada contexto, incluindo o do seu projeto.

🕳️ Esparsidade (Sparsity)

A maioria das células da matriz é NaN. Com poucos ratings compartilhados, a similaridade entre usuários se torna instável, pois dois usuários que avaliaram apenas um item em comum podem parecer muito similares por acidente. Quanto mais esparsa a matriz, menos confiável o UserKNN.

❄️ Cold Start

Novo usuário: sem histórico de ratings, não é possível calcular similaridade com pessoa alguma.

Item: sem avaliações, nunca aparece nas recomendações.

Esse é o ponto cego de toda filtragem colaborativa pura, e por que sistemas reais combinam filtragem colaborativa com filtragem por conteúdo.

📐 Escalabilidade

Calcular similaridade entre todos os pares de usuários tem custo O(n²). Com 10 milhões de usuários, isso é inviável em tempo real. SVD e embeddings neurais existem parcialmente por causa desse problema; eles pré-computam representações mais compactas.

Para o dataset sintético de hoje

No notebook, vocês vão gerar um dataset com ~60% de esparsidade. É suficiente para sentir o problema sem travar o algoritmo. Na dinâmica, peço que reflitam sobre o seguinte: se a esparsidade fosse 95%, o que mudaria nos resultados do UserKNN?

Métricas de similaridade entre usuários

A qualidade do UserKNN depende diretamente da métrica de similaridade escolhida. Cada uma tem premissas diferentes sobre o que significa "dois usuários serem parecidos".

Métrica Fórmula simplificada Quando usar Limitação
Cosseno cos(u,v) = (u·v) / (‖u‖·‖v‖)
trata NaN como 0		 Ratings com escala bem definida. Bom ponto de partida. Padrão do sklearn. Médio Ignora viés de usuário (quem sempre dá 5 vs. quem dá 3 para o mesmo nível de gosto)
Pearson corr(u,v) = cov(u,v) / (σu·σv)
apenas itens co-avaliados		 Melhor para ratings explícitos. Remove viés de escala (normaliza pela média). Sensível Instável com poucos itens em comum; correlação espúria com apenas 2–3 ratings sobrepostos
Jaccard J(u,v) = |A∩B| / |A∪B|
conjuntos de itens avaliados		 Dados implícitos (binários: avaliou ou não). Ignora o valor do rating. Simples Descarta a magnitude dos ratings; não distingue 1 estrela de 5 estrelas

UserKNN: do vizinho ao rating previsto

O pipeline completo em quatro passos. A fórmula de predição com ajuste de viés é o coração do algoritmo. É o que diferencia um recomendador ingênuo de um que lida com usuários que tendem a dar notas altas ou baixas sistematicamente.

Passo 1

Similaridade

Calcule sim(u, v) para todos os pares. Resultado: matriz n×n de similaridades.

Passo 2

Vizinhos

Para o usuário-alvo u, encontre os k mais similares que avaliaram o item i.

Passo 3 · core

Predição

Média ponderada dos ratings dos vizinhos, ajustada pelo viés de cada um.

Passo 4

Top-N

Preveja ratings para todos os itens não avaliados. Retorne os N maiores.

r̂(u,i) = r̄_u +   Σv ∈ N(u)  [ sim(u,v) · (r(v,i) − r̄_v) ]  /  Σv ∈ N(u) |sim(u,v)|

r̄_u = média dos ratings do usuário-alvo
r̄_v = média dos ratings do vizinho v
N(u) = conjunto dos k vizinhos mais similares que avaliaram i
O ajuste (r(v,i) − r̄_v) remove o viés: se um vizinho sempre dá notas altas, desconta-se essa tendência.

SVD: descobrindo padrões que não estão escritos nos dados

Em vez de perguntar "quem é parecido com você?", o SVD pergunta "quais fatores ocultos explicam todos os ratings observados?". Esses fatores são os fatores latentes — dimensões abstratas que capturam preferências que ninguém nomeou.

O que são os fatores latentes?

São dimensões abstratas que o algoritmo descobre sozinho nos dados. Para filmes: podem capturar "filme de ação vs. drama", "produção de Hollywood vs. cinema independente", "anos 80 vs. contemporâneo". Você não define os fatores — o SVD os infere. O predito r̂(u,i) = U[u] · Vᵀ[:, i] é o produto interno dos vetores latentes.

Por que truncar (TruncatedSVD)?

A decomposição completa de R tem tantos fatores quanto a menor dimensão da matriz. A maioria carrega ruído. Com d fatores (d = 10–50 é comum), capturamos os padrões principais e ignoramos o ruído. Isso funciona como uma regularização implícita, reduzindo o overfitting à esparsidade.

NaN antes do SVD

SVD não aceita NaN. A estratégia mais simples: preencher com a média do usuário antes de decompor. Alternativas mais sofisticadas (como o SVD++ do Netflix Prize) aprendem a preencher os valores como parte do treinamento.

UserKNN vs. SVD: quando usar cada um

Nenhum dos dois é universalmente melhor. A escolha depende do tamanho do dataset, da importância de interpretabilidade, da esparsidade e do contexto do negócio. Isso é o que vocês vão verificar empiricamente no notebook de hoje.

Critério UserKNN SVD
Interpretabilidade Alta "Você e Ana avaliaram 8 itens em comum e têm 0.87 de similaridade" Baixa Fatores latentes não têm nome nem interpretação direta
Esparsidade Sensível Com > 95% esparso, poucos vizinhos úteis; predições instáveis Robusto SVD lida bem com esparsidade — é justamente onde brilha
Escalabilidade Limitado O(n²) para similaridade. Inviável > 100k usuários sem aproximações Escalável TruncatedSVD é eficiente; retreinamento periódico, não em tempo real
Cold Start Problema Usuário novo sem ratings → sem vizinhos → sem recomendação Problema Igualmente afetado; novo usuário não está na decomposição
RMSE típico Moderado Melhor em datasets densos e domínios de nicho Melhor Geralmente supera UserKNN em datasets esparsos e grandes

Debrief: o que costuma aparecer na primeira implementação

Três padrões que aparecem em toda turma que implementa UserKNN e SVD pela primeira vez em dados próprios.

1 · RMSE do UserKNN altíssimo

Causa mais comum: esparsidade muito alta no dataset, com poucos itens co-avaliados entre vizinhos. A predição acaba usando a média do usuário-alvo sem contribuição real dos vizinhos. Solução: gere mais avaliações no dataset ou reduza k.

2 · SVD e UserKNN recomendam os mesmos itens

Sinal de que o dataset tem pouca variação — os perfis de usuário gerados pela LLM são homogêneos demais. Peça à LLM para injetar mais diversidade nos perfis. Quando funcionar direito, os dois algoritmos devem ter overlap parcial, não total.

3 · SVD com variância < 40%

O dataset não tem estrutura latente suficiente para o SVD capturar. Pode ser dataset pequeno ou muito esparso. Reduza d (número de fatores) para min(n_usuários, n_itens) − 1 e verifique se o gráfico de variância acumulada tem cotovelo visível.

A pergunta certa para o relatório

Não é "qual algoritmo tem menor RMSE?", e sim "qual algoritmo faz mais sentido para o contexto do projeto?" Um RMSE 0.02 melhor não justifica perder interpretabilidade se os stakeholders precisam entender o porquê da recomendação. Isso é o que separa um bom relatório de uma lista de métricas.

Checagem 1 de 2 · Similaridade e viés

Ana sempre dá notas entre 4 e 5. Bruno sempre dá notas entre 1 e 2. Ambos deram as mesmas notas relativas para os filmes que assistiram juntos — ou seja, quando Ana dá 5, Bruno dá 2; quando Ana dá 4, Bruno dá 1. Qual métrica de similaridade captaria melhor essa afinidade de preferência?

  • Similaridade do cosseno, pois compara os vetores de ratings diretamente sem normalização
  • Correlação de Pearson, pois normaliza os ratings pela média de cada usuário e detecta a relação linear entre as avaliações
  • Índice de Jaccard, pois considera apenas se os usuários avaliaram os mesmos itens, independentemente das notas
  • Distância euclidiana, pois mede a diferença absoluta entre as notas de cada item

Checagem 2 de 2 · UserKNN vs. SVD na prática

Um sistema de recomendação médica sugere artigos científicos para pesquisadores. O dataset tem 500 pesquisadores, 2.000 artigos e 98% de esparsidade. Os gestores de saúde precisam entender por que um artigo foi recomendado para cada pesquisador. Qual abordagem é mais adequada nesse contexto?

  • SVD com 50 fatores latentes, pois lida melhor com alta esparsidade e terá RMSE menor
  • SVD com 5 fatores latentes, pois a baixa dimensionalidade tornará os fatores interpretáveis
  • UserKNN com k pequeno e Pearson, pois a explicação "pesquisadores com perfil similar a você também leram estes artigos" é auditável e compreensível pelos gestores
  • Nenhum dos dois, pois 98% de esparsidade invalida qualquer filtragem colaborativa
